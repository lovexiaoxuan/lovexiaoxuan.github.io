A platform based on ZYNQ for ADDA systems

争对2024电子设计竞赛国赛基于ZYNQ搭建的ADDA硬件平台

  电赛结束也快一个月的时间了，很早之前就有过自己搭建一个硬件测试平台的想法，但一直由于时间和技术问题没有得以实施。在用余佬（TQ-V85Sn）的板子打完省赛后，我想着自己也该花点时间研究一下各个芯片的细节和可拓展性，最终这篇博客应运而生（当然后面还要无限次修改）。其实这早就不该是自己的第一篇博客，但因为本人懒散一拖再拖先前很多项目都只能躺在硬盘的某个分区慢慢吃灰了，既然决定开始记录那就从这个项目开始吧，有句话怎么说来着：

凡是过往，皆为序章。

芯片选型

  主控芯片：怎么说，好歹也是最后一次打国赛了，当然要选XC7Z020-2CLG400l。选的是野火的核心板（转了半天发现最便宜的020核心板就是这一块）。
  ADC芯片：之前一直用的AD9269，80M采样率的上限，16比特的采样精度，CMOS输出其实已经由于大部分市场上已有的ADC模块选用的芯片，但毕竟是自己重新制作嘛，还是挑战一下更高采样率的ADC，以及稍有不同的高速协议LVDS，最终经过挑选和DAC精度匹配采样率高达250Msps的AD9643（虽然电赛DA挑战了时序最高只能跑到200M，但250M值得挑战）14比特采样精度。
  DAC芯片：貌似采样率300M以上就奔着G去了，美信的芯片咱也用不起，当然初体验以及性价比最高的还得是AD9744，但是系统既然换就要换全套，经过一系列价格性能的筛选AD9746刚好和AD的采样率很好的匹配上。
  PLL芯片：之前用的Si5351采用I2C通信接口，基本上可以实现150M以下频率全覆盖，200M频率便是极限，由于ADDA的时钟由PLL专门提供最为理想，ZYNQ内部PLL不进行输出，则Si5351要同时给AD，DA，ZYNQ提供参考时钟，所以5351最不能接受的性能是

  AD9552是一款比较符合设计要求的PLL芯片，虽然不能实现频率连续全覆盖，但在50~250M的频率范围内重要的频率均可提供，提供LVDS协议，LVPECL协议和CMOS输出
  ADDA驱动芯片：目前考虑使用LTC6419（单端转差分的ADC驱动器）代替ADC前级的巴伦来改善AD的低频采用能力（尚待开发…）

芯片开发细节

  1. XC7Z020-2CLG400l
    • ZYNQ芯片的启动过程和复位分析
        此部分详情见ug585手册 Chapter 6:Boot and Configuration
        ZYNQ的启动分为两种情况：secure Boot Mode 和 non-secure Boot Mode，本系统仅对 non-secure Boot Mode 进行分析：

      JTAG always implies a non-secure environment.

      1. 系统上电启动执行
          系统上电后，复位引脚 PS_POR_B（该引脚连接在BANK500上）被置位，此刻 ZYNQ 硬件立马去采样一组称作 Boot Strap Pins 的管脚，这组管脚告诉硬件以什么方式启动(QSPI Flash、NAND Flash、NOR Flash、SD Card)，以及是否开启PLL等，如果 Boot Mode 不是 JTAG，则 BootROM 配置所选择的 boot device 控制器。
          PS 端首先执行 BootROM 上的启动代码，这段代码是直接在 On Chip ROM（OCM）中 XILINX 烧刻进去，无法看到而且也无法修改，所以 BootROM 是上电后，APU 执行的第一段代码就是 BootROM 的这部分， ZYNQ-7000 是双核 Cortex-A9，两个 CPU 核心，BootROM 由 CPU_0 来执行，此刻 CPU_1 处于 WFE 等待状态。BootROM 的主要任务是配置系统，BootROM读取存储器中的程序代码，并将FSBL拷贝到OCM（On-chip memory）里 ， 这个被拷贝到片上 RAM 执行的程序就来自于我们要制作的文件——— BOOT.bin (如下图)。或者，可以在 non-secure Boot Mode 中直接从 Quad-SPI 或 NOR 设备执行 FSBL/User Code，即片上运行代码。

        🐖：由于BootROM可以从QSPI/NAND/NOR Flash/SD Card读取数据，则BootROM中应该含有其相关的标准驱动和标注访问指令。特别的该核心板采用 QSPI Flash 启动方式，连接在BANK500的PS_MIO1~PS_MIO6。
          BootROM 采集完 Boot Strap Pins 后呢，初始化对应的外设控制器，在启动设备上寻找相应的 FSBL 的过程中，BootROM 会去启动设备上不断的寻找这个 BootROM Header，以便完成接下来的工作交接。所以启动设备在整个启动过程中会提供 BootROM Header 和 FSBL/User Code。
          在 BootROM 找到 BootROM Header后，系统的控制权交给FSBL，如果携带了bit流文件，则继续配置 PL 端资源，最后加载 BootLoadr。
